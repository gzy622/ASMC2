## 📋 执行摘要

本报告对 **AssignmentCheck2** 项目进行了全面的技术债审查。该项目是一个面向教师的课堂作业登记工具，采用纯前端技术（HTML/CSS/JavaScript）开发，无需构建即可在浏览器中运行。

**总体评估**：项目代码质量良好，但存在一些需要关注的技术债问题，主要集中在**代码组织**、**样式管理**和**测试覆盖**方面。

## 🔍 详细技术债清单

### 1. 代码组织类

#### 1.1 全局命名空间污染 ⚠️

**位置**：`app.js`, `actions.js`, `modal.js` 等核心文件

**表现形式**：
```javascript
// 所有模块都直接暴露在全局作用域
const State = { ... };
const Actions = { ... };
const Modal = { ... };
```

**通俗解释**：就像把所有文件都堆在桌面而不是放进文件夹。虽然找起来方便，但随着项目变大，变量名冲突的风险会增加。

**潜在风险**：
- 第三方库引入时可能发生命名冲突
- 难以进行代码分割和按需加载
- 不利于单元测试的隔离性

**建议措施**：
- 考虑使用 ES6 模块系统（import/export）
- 或采用简单的命名空间模式（如 `App.State`, `App.Actions`）

---

#### 1.2 文件职责边界模糊 ⚠️

**位置**：`app.js` 与 `actions.js`

**表现形式**：
- `app.js` 包含状态管理、数据持久化、业务逻辑和视图渲染
- `actions.js` 和 `action-views.js` 分离但仍有交叉

**通俗解释**：就像一个员工同时负责财务、人事和销售，虽然初期效率高，但随着业务增长会变得难以管理。

**具体案例**：
```javascript
// app.js 中混合了太多职责
const State = {
    // 状态管理
    list: [], roster: [], data: [],
    // 数据解析
    parseRoster() { ... },
    // 持久化逻辑
    save() { ... },
    // 视图相关
    view: { init() {}, render() {} }
};
```

**建议措施**：
- 将数据层（Model）、视图层（View）、控制层（Controller）进一步分离
- 或明确采用 Flux/Redux 风格的状态管理模式

---

#### 1.3 工具函数分散 ⚠️

**位置**：`core.js` 与各个模块

**表现形式**：
- 通用工具（`ColorUtil`, `IdGenerator`, `Validator`）集中在 `core.js`
- 但部分模块仍有内联的工具函数

**通俗解释**：工具就像公司的公共设施，应该统一放置，而不是每个办公室都备一套。

**建议措施**：
- 建立统一的工具库目录结构
- 明确工具函数的使用规范

---

#### 1.4 魔法数字和字符串分散 🔢

**位置**：多个文件

**表现形式**：
```javascript
// modal.js
FULL_ENTER_MS: 220,
FULL_EXIT_MS: 160,
PAGE_EXIT_MS: 160,
LOADING_MASK_FADE_MS: 90,

// scorepad.js
setTimeout(() => { ... }, 120);
```

**通俗解释**：代码中散落着各种数字（220、160、90），就像食谱里写"加适量的盐"——对新手来说很难理解其含义。

**建议措施**：
- 将动画时长、延迟等常数提取到统一的配置文件中
- 添加注释说明数值的业务含义

---

### 2. 样式管理类

#### 2.1 CSS 文件过大 🎨

**位置**：`styles.css`

**数据**：
- 文件大小：约 20KB+
- 代码行数：约 1200+ 行

**通俗解释**：就像把所有衣服都塞进一个抽屉，找起来很困难。

**具体表现**：
- 包含全局样式、组件样式、动画、响应式布局
- 缺乏模块化组织
- 部分选择器可能已废弃但未清理

**建议措施**：
- 按组件拆分 CSS 文件（如 `modal.css`, `scorepad.css`）
- 或使用 CSS-in-JS 方案
- 建立样式命名规范（BEM 等）

---

#### 2.2 样式与 JavaScript 耦合 🎨

**位置**：`action-views.js`, `modal.js`

**表现形式**：
```javascript
// 在 JS 中直接操作样式
nav.style.cssText = 'display:flex;flex-direction:column;flex:1;min-height:0';
preview.style.background = `linear-gradient(160deg, ${s}, ${e})`;
```

**通俗解释**：把装修方案写进了家具说明书，想换装修风格就得重新写说明书。

**建议措施**：
- 优先使用 CSS 类切换而非直接操作样式
- 将动态样式值通过 CSS 变量传递

---

#### 2.3 响应式断点不统一 📱

**位置**：`styles.css`

**表现形式**：
```css
@media (max-width: 680px) { ... }
@media (max-width: 640px) { ... }
@media (max-width: 400px) { ... }
```

**通俗解释**：就像门的尺寸标准不统一，有的 80cm，有的 75cm，安装时容易出错。

**建议措施**：
- 定义统一的断点变量
- 建立响应式设计规范文档

---

### 3. 测试覆盖类

#### 3.1 测试覆盖率不足 🧪

**位置**：`tests/` 目录

**数据对比**：

| 模块 | 代码行数（估算） | 测试覆盖 | 覆盖率 |
|------|----------------|---------|--------|
| core.js | ~400行 | core.test.js | 🟢 良好 |
| app.js | ~800行 | state.test.js | 🟡 部分 |
| actions.js | ~600行 | app.spec.js | 🔴 有限 |
| modal.js | ~500行 | modal-progressive.test.js | 🟡 部分 |
| scorepad.js | ~300行 | e2e_scorepad_fast_ten.spec.js | 🔴 有限 |

**通俗解释**：就像新车只测试了发动机，没测试刹车和转向灯。

**建议措施**：
- 为核心业务逻辑增加单元测试
- 增加集成测试覆盖关键用户流程
- 设置测试覆盖率阈值（如 70%）

---

#### 3.2 E2E 测试依赖浏览器 🌐

**位置**：`tests/*.spec.js`

**表现形式**：使用 Playwright 进行端到端测试

**潜在风险**：
- 测试运行时间较长
- 环境依赖较重（需要安装 Chromium）
- 测试稳定性受浏览器行为影响

**建议措施**：
- 保持 E2E 测试聚焦核心用户流程
- 更多使用单元测试覆盖业务逻辑

---

### 4. 架构设计类

#### 4.1 状态管理复杂度 ⚙️

**位置**：`app.js` 中的 `State` 对象

**表现形式**：
```javascript
const State = {
    // 基础状态
    list: [], roster: [], data: [], curId: null,
    // 功能开关
    mode: 'name', scoring: false, animations: true,
    // 缓存相关
    _cacheVersion: 0, _metricsCache: new Map(),
    // 脏标记系统
    _dirtyData: false, _dirtyList: false,
    _gridDirtyFull: true, _gridDirtyStudentIds: new Set(),
    // 定时器
    _persistTimer: 0, _draftTimer: 0,
    // ... 更多属性
};
```

**通俗解释**：就像一个控制面板上有太多按钮，操作员容易按错。

**建议措施**：
- 将状态分为"业务状态"和"UI 状态"
- 考虑引入轻量级状态管理库（如 Zustand、Valtio）

---

#### 4.2 数据持久化与业务逻辑混合 💾

**位置**：`app.js` 中的 `save()` 方法

**表现形式**：
```javascript
save({ render = true, immediate = false, dirtyData = true, 
       dirtyList = false, asgListChanged = false, 
       normalizeMode = 'all', targetAsgId = null, 
       invalidateDerived = true } = {}) {
    // 数据验证
    // 状态更新
    // 持久化逻辑
    // 视图渲染触发
}
```

**通俗解释**：把存文件、算数据、刷新屏幕三件事绑在一起做，想改其中一个就得全改。

**建议措施**：
- 采用 Repository 模式封装数据持久化
- 使用事件机制解耦状态变更与视图更新

---

### 5. 文档注释类

#### 5.1 部分复杂逻辑缺少注释 📝

**位置**：`modal.js` 的动画系统

**表现形式**：
```javascript
// 复杂的动画时序控制
armPointerGuard(ms = 320) {
    clearTimeout(this._pointerGuardTimer);
    this.el.style.pointerEvents = 'none';
    this._pointerGuardTimer = setTimeout(() => {
        this._pointerGuardTimer = 0;
        if (this.isOpen && !this.isClosing) this.el.style.pointerEvents = '';
    }, ms);
}
```

**通俗解释**：就像电路图没有标注，维修时很难理解设计意图。

**建议措施**：
- 为复杂算法添加实现注释
- 记录设计决策和权衡考虑

---

#### 5.2 缺少架构文档 📚

**表现形式**：
- README 主要关注使用说明
- 缺少架构设计文档
- 模块间依赖关系不清晰

**建议措施**：
- 添加 ARCHITECTURE.md 文档
- 绘制模块依赖图
- 记录关键设计决策

## 📋 总结

### 项目优势 ✅

1. **功能完整**：覆盖了作业登记的核心场景
2. **代码规范**：命名清晰，结构相对合理
3. **测试基础**：已建立测试框架和基础用例
4. **用户体验**：注重动画和交互细节

### 需要关注 ⚠️

1. **样式管理**：CSS 文件过大，需要模块化
2. **测试覆盖**：核心业务逻辑测试不足
3. **架构边界**：模块职责需要进一步明确

### 总体评价

该项目是一个**质量良好的中小型前端项目**。技术债主要集中在"规模化管理"方面——随着功能增加，当前的简单架构可能会成为瓶颈。建议在下次大版本迭代时，优先解决代码组织和样式管理问题。
